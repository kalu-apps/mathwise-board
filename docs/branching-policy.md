# Политика ветвления (main-only)

## Цель

Ускорить поставку в прод: одна основная ветка `main` без промежуточной `next`.

## Базовая схема

- `main` — единственная рабочая и релизная ветка.
- `next` — выведена из эксплуатации.

## Поток работ

1. Вносить изменения в `main`.
2. Пушить в `main`.
3. Деплоить из `main`.

Опционально для рискованных задач:

1. Создать ветку от `main` (`fix/*`, `feature/*`, `refactor/*`, `perf/*`, `chore/*`, `docs/*`).
2. Открыть PR в `main`.
3. После merge деплоить `main`.

## Жёсткие правила

- Любая актуальная версия кода должна быть в `main`.
- Работу через `next` не использовать.
- Флоу `next -> main` и `main -> next` запрещены.
- Одна правка не реализуется вручную в двух ветках.

## Рекомендация по GitHub protection

Для `main`:

- включить required status checks (`Quality Gates`, `Branch Flow Guard`);
- ограничить force-push;
- включить Include administrators.
